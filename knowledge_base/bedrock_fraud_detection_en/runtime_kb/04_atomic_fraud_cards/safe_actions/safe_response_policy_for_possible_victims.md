# Safe response policy for possible victims

## Pattern summary
User-facing steps for people who may have interacted with a scam, shared data, installed software, or sent money.

## Core deception
The user needs containment and verification without being blamed or asked for more secrets.

## Common channels
- all

## Red flags
- User says they clicked a link, entered data, shared code, sent money, or installed app
- Ongoing call/chat pressure
- Account, card, or device may be compromised

## Scam examples
- I already entered my card on the page.
- I gave the code to the caller.

## Legitimate contrast
If the user only received a suspicious message and did not interact, focus on prevention and verification rather than incident response.

## What NOT to classify as fraud automatically
- Do not assume loss occurred without user interaction
- Do not ask user to paste full card number, OTP, password, or ID documents

## Safe user actions
- Stop communicating with the suspected scammer
- Contact bank/service through official channel
- Change passwords/revoke sessions if credentials were entered
- Preserve screenshots, URLs, numbers, and transaction IDs

## Runtime model instruction
Use this card to produce safe, calm next steps. Never request secrets. Prioritize bank contact, account security, device containment, and evidence preservation.
